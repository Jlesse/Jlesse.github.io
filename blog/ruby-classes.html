Classes, Objects, and The Dude

Ruby is an object-oriented programming language. You may be wondering, what does that even mean? In the most basic sense, it means Ruby's programming focus is on objects which possess attributes and methods. Attributes are information about the object's state. If you create a car as a Ruby object, it might have an attribute specifying the color. Attributes are often thought of as nouns; methods, on the other hand, are thought of as verbs. For example, using the same car analogy, the car object might have a method named drive, which instructs the car object to begin driving.

Classes give us what we need to create objects. Classes are like blueprints, you reuse the class structure to create new objects that are unique but share the same methods and attributes. Some objects may be made from the same class, but when an attribute is modified in one object it does not change the attribute in another object made from the same class (unless the attribute is a global variable or class variable). That is part of the beauty of OOP (object-oriented programming) which is known as Encapsulation. Data and functions are packed into a single unit, and you have control over what can be seen, changed, or used.

Lets look at an example. Lets create a Person class. What kind of attributes should the person class have? Sure there could be an infinite amount, but lets try keep this simple. For our purposes a person has an age, a name, a location, and they are either hungry or not hungry. Lets see what that might look like in ruby.

Here we have declared a class named Person, and written an initialize method. Inside the initialize method are all of the attributes of the class. These attributes are called instance variables. They exist within a single instance of the object. Instance variables must begin with an @ symbol. An initialize method is used to start a program off in a specific state and create all of the instance variables. Its not required, but 99% of the time you will want to use an initialize method.Values can be passed into the initialize method to assign to the instance variables. In this case we can pass in age, name, location, and hunger. We are only using instance variables in this example because we want to keep the access of the variables contained in the object. If we used global or class variables, they would be available outside of the object or class. Its a good idea to use an instance variable if the variable holds important information about the state of the object, or the variable is used in the class in multiple methods. If the variable is only used inside one method, its probably best to define that variable inside that method. In the case the variable is defined inside a method, it is a local variable, and can only be accessed inside that method. The variables scope (where it can be accessed) would be the method body.

Now lets create some methods that allow this person do things. People like doing all kinds of things, but lets keep it simple. People eat, sleep, and work. Hopefully most people do more than that, but thats all that the people from our Person class do. What would this look like?

Here we have defined 3 methods. Eat, work, and sleep. They are all instance methods becuase they are defined inside the class, and an instance of the class may use them. Eat takes a parameter named food_item. If the person is hungry(if hungry is true), then the person will eat and say "Yum yum, I was hungry. That was good (whatever food_item was)!", and then the hunger instance variable will be set to false since the person is not hungry anymore. If they are not hungry (If hungry is false), they will say "I am not hungry!". Work just makes the person say "work work!". In the sleep method, If the person is hungry they will "I am too hungry to sleep! feed me!". If they are not hungry they will say puts "ZZZZZZZ(snore)". A pretty simple person, but hey, it works for our purposes. You can see how these methods are like actions. They allow the person to do things. The instance variables might change based on these actions, like the hunger variables for example.

Now that we have defined our methods, lets call them.

when say dude = Person.new(25,"theDude", "Chicago", true), we have created a new object of type person. The dude objects age is 25, its name is "theDude", the location is "Chicago", its hunger status is true. when we call dude .work, the dude say "work work!" when we try to sleep, he complains because he is hungry. when we feed him eggs he says he likes them. when we try to make him sleep again, he goes to sleep because he is no longer hungry.

The dude object posses all of the methods and attributes of the Person class. we call them with object-name.method-name. Thats what you see me do when I say dude.eat for example.
